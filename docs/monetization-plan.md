# 閻魔庁ONLINE 住民課金(月額1,800円)設計+実装引き継ぎガイド

作成: 2026-07-23 / 前提資料: retention-plan.md・growth-design.md・chatgame-design.md・enmacho_growth_model.xlsx

---

## 1. 何を売るか — 「強さ」ではなく「快適・見た目・応援」

閻魔庁ONLINEはPvPが無い協力型なので、多少の数値優位が他人の損にならない。それでも
「チャットゲーム化」を軸にする以上、**友達関係を壊す課金だけは絶対に避ける**。守る3原則:

1. **無料で完全体験できる。** 全章・全職・全スキル・全装備は無料で到達可能。住民は「速くて快適」なだけ
2. **住民がPTにいると周りも得をする。** 妬まれる課金ではなく、歓迎される課金にする
3. **数値優位は合計でも+15%程度まで。しかも追いつける。** 差は「速度」であって「到達可能性」ではない

世界観の名前案: **「定住許可証」**。死後世界の役所で住民登録をする、という設定がそのまま使える。
(仮住まいの魂=無料プレイヤー、正式に住民登録した魂=課金者。ゲームの世界観と完全に一致する)

## 2. 定住許可証の内容案(月額1,800円)

### 快適系(いちばんの購買動機。強さではなく時間を買う)

- **どこでも高札**: メニューから御用の受注・納品ができる(町へ戻る往復が消える)
- **渡し船の無償化+町間の瞬間移動**(章内のみ)。移動時間の短縮
- **倉庫拡張**: 所持枠+50(将来の露店・素材増に備えた基盤も兼ねる)
- **復活地点の選択**: 死亡時、賽の石碑だけでなく各町の石碑から復帰先を選べる

### 配給系(毎月の実利。数値はここに集約し、+15%原則の中に収める)

- **月初の住民配給**: 甘露水L×20、韋駄天・修羅各×5、**巻物のかけら×30(=巻物3枚分)**
- **恩賞(ログインボーナス)が2倍**(時給5%→10%。設計表の比率パラメータで管理)
- **休眠徳**: ログインしなかった日数に応じて「休眠徳」が溜まり、次回の狩りでXP+50%
  (上限3日分)。忙しい社会人が友達に追いつける仕組み。**無料でも1日分は溜まる**(住民は3日分)

### 見た目・記名系(ステータスにならない自慢。原価ゼロで満足度が高い)

- 住民専用の**衣装染め**(既存スプライトの色相変え=tint技術は実装済み)
- 名前の**称号枠**(「定住者」+実績称号)とチャット文字色
- 合掌エモートの専用エフェクト(蓮の花が咲く等)

### ソーシャル系(原則2の実装。住民は「いてくれると嬉しい人」になる)

- **住民の焚いた篝火**: 住民は野外に小さな篝火を置ける(30分)。**そばにいる全員**(無料含む)に
  小休憩ボーナス(HP/MP回復速度up)。住民が集いの中心になる
- **お裾分け枠+2**: デイリーのフレンドへの贈り物が1人→3人に
- 講の作成費無料+講の週間目標達成時、住民がいる講は恩賞に消耗品が1品追加(講全員に)

### 入れてはいけないもの(明文化しておく)

- 強化成功率アップ、住民限定装備・スキル、XPの恒常大幅ブースト(+50%等)
- 募集ポップアップやマッチングでの優先表示(出会いの装置は平等が絶対条件)
- ガチャ・ランダム商品全般(信頼で成立するゲームに賭博を持ち込まない)

## 3. 「このままの設計でよいのか」— 影響チェックの結論

**結論: 骨格はこのままでよい。調整は3点だけ。**

1. **バランスは常に無料プレイヤー基準で設計する。** 設計表の想定時間・報酬はすべて無料基準の
   まま維持し、住民は結果的に1〜2割速いだけ、とする。住民基準で設計し始めると無料が苦行になり、
   新規流入(retention-plan.md)が死ぬ。これが最重要
2. **巻物の週間供給を2本立てにする。** 設計表「強化新案」シートの週間入手目安を
   無料5枚/住民8枚(配給3枚込み)として読む。+9到達が無料3.5週/住民2.2週——差はあるが
   どちらも健全圏。シートの入力セルを書き換えれば即検証できる
3. **課金導線は定着の後に置く。** 初回10分〜第1章クリアまでは課金要素を一切見せない。
   定住許可証の案内は「2章到達(転職)時」が最初。順序は 定着→愛着→課金 であり逆にしない

既存設計との相性が良い点も確認済み: デイリー・恩賞・伝授料が全部「時給比」で設計してあるため、
住民特典も比率1つで表現でき、インフレしない。休眠徳はチーム前提の設計(友達と徳位が離れない)
とも噛み合う。

## 4. 売上の目安(そろばん)

転換率は買い切りでない月額制の個人ゲームで2〜5%が相場感。控えめに3%で:

| MAU(月間プレイヤー) | 住民数(3%) | 月商 |
|---|---|---|
| 300 | 9 | 16,200円 |
| 1,000 | 30 | 54,000円 |
| 3,000 | 90 | 162,000円 |
| 10,000 | 300 | 540,000円 |

現在のインフラ(Cloudflare Pages+Workers+Supabase東京+Resend)は当面ほぼ無料枠なので、
MAU 300で黒字化する構造。「ユーザーが増えたら」の判断ラインはMAU 500〜1,000到達が目安。
GA4のMAUがそのまま判断材料になる(計測を先に入れるべき理由がここにもある)。

## 5. 決済の技術メモ(実装時期が来たら)

- **Stripe Checkout(サブスクリプション)+ Supabase** が本命。Webhook を Supabase Edge Function で
  受けて `entitlements` テーブルを更新、クライアントはRLS越しに自分の住民状態を読むだけ
- アプリストアを介さないブラウザゲームなので手数料はStripeの3.6%のみ(ストア30%が無い)
- 必要な事務: 特定商取引法に基づく表記ページ、プライバシーポリシーへの決済情報の追記、
  解約導線(ワンクリック解約をケチらない。信頼が商品)
- ゲーム内通貨の販売はしない(モンを直接売らない)。前払式支払手段の供託義務など
  資金決済法まわりの重い論点を最初から回避できる。この項は参考情報であり、開始前に
  税理士・法律の専門家への確認を推奨
