# 新規流入が定着する仕組み — 設計書

対象: 閻魔庁ONLINE (enmacho.com) / 作成: 2026-07-23
目的: 新しく来たプレイヤーが「最初の10分で辞めない」「翌日もう一度来る」ようにする。
この文書はそのまま `docs/retention-plan.md` としてリポジトリに入れられる形で書いてある。
実装は Claude Code(Mac側)担当。各施策に対象ファイルと発火箇所を明記した。

---

## 1. 現状診断(コード調査の結果)

新規プレイヤーのファネルと、各段の現状:

```
流入(検索・告知) → TOP(index.html) → play.html → 入庁画面 → 初戦闘 → オキナ会話
 → 試練の洞窟 → オニオウ → 竜の洞窟 → ゴウリュウ → 第二章
```

強み(すでに良い点):

- 入庁は実質1クリック。名前未入力でも「ナナシ」で開始でき、登録も強制しない。摩擦は最小
- 死亡ペナルティゼロ(play.html:3255-3263)。初見が死んでも失うものがなく、心理的に優しい
- 右上の常設ゴールHUD(play.html:7667-7685)が「まずはオキナの話を聞こう」を出している
- オートセーブが堅牢(離脱時保存あり)。進行が消える系の離脱は起きにくい

離脱要因(重要度順):

1. **どこで離脱しているか計測できていない。** ゲーム内のGA4イベントは `game_start`(play.html:8949)の1つだけ。初戦闘・初クエスト・死亡・レベルアップ・洞窟到達のイベントがなく、序盤ファネルが見えない
2. **操作説明が初回タイトル限りで、二度と見られない。** howtoList(play.html:1183-1189)は再入庁時非表示(9662-9664)。設定画面にもヘルプ項目がない
3. **最初の一歩が「オキナに話しかける」ことに依存。** オキナに話すまで `quests.trial=0` のまま(4266)。初見はどのNPCがオキナか分からず、開始時のログ3行(8955-8957)は流れて消える
4. **翌日戻ってくる理由がない。** デイリー要素・連続入庁の恩賞・時限イベントの予告表示がない。ゴウリュウの30分復活は序盤プレイヤーには無関係
5. **魂の資質5ポイントの割り振り**をログ1行で促すのみ。気づかず弱いまま戦い「難しい」と感じて辞める恐れ
6. TOP→playの導線にメール獲得の仕組みが弱い。Resend連携(account.js:771-775)はあるのに、辞めた人を呼び戻す手段として使われていない

## 2. 施策一覧(3層構造)

### C層: まず「見える化」— GA4ファネルイベントの追加 【最優先・工数小】

理由: 母数が少ない今こそ、1人1人の離脱地点が貴重なデータになる。以降のA/B層の効果測定もこれが前提。

実装: play.html内の各所へ `window.enmachoAnalytics?.track(名前, パラメータ)` を1行ずつ追加するだけ。
既存の `track()`(online/analytics.js:114-124)は同意済みのときだけ送る安全設計なので、そのまま使える。

| イベント名 | 発火箇所(目安) | パラメータ |
|---|---|---|
| `tutorial_step` | 下記A-1の各ステップ達成時 | `step`(1〜5) |
| `first_kill` | 初めて敵を成仏させた時(セーブにフラグ追加) | `mob` |
| `quest_accept` / `quest_complete` | 高札受注・達成処理 | `quest_id` |
| `level_up` | 徳位上昇時 | `level` |
| `player_death` | 死亡処理(play.html:3255付近) | `zone`, `level` |
| `zone_enter` | ゾーン切替(試練の洞窟・竜の洞窟・輪廻大陸) | `zone` |
| `stat_allocate` | 資質を初めて割り振った時 | なし |
| `account_register` / `cloud_save` | 魂籍登録完了・クラウド保存(account.js) | なし |

注意:
- 個人を識別できる値(名前・メール・チャット)は入れない(docs/analytics-setup.mdの方針を維持)
- イベントを増やしたら docs/analytics-setup.md のイベント一覧も更新する
- GA4側で「game_start → first_kill → quest_accept → zone_enter(cave)」のファネル探索レポートを作る(これはユーザー作業。手順は後述の付録)

### A層: 初回10分の離脱を減らす(オンボーディング)

**A-1. 着任チェックリスト(段階式ガイド)** 【効果大・工数中】

初回プレイ時のみ、右上ゴールHUD(#quest, play.html:1029)を5段のチェックリストに差し替える:

1. 地面をタップして移動する
2. 迷える魂(ガキ)を1体成仏させる
3. 川辺のオキナと話す
4. メニュー(I)で魂の資質を割り振る
5. 高札でお勤めを受ける

- 各達成でチェック音+「✓」表示、全達成で報酬(甘露水S×3+50モンなど軽いもの)
- 進行はセーブに `tutorial:{step}` として保存(SAVE_VERSION=15→16に上げ、旧セーブは達成済み扱いで移行)
- 各ステップ達成時に `tutorial_step` イベント送信(C層と連動)
- 既存の `quests.trial` 進行とは独立させ、既プレイヤーには表示しない

**A-2. オキナへの道標** 【効果中・工数小】

チェックリストのステップ3が現在の時、オキナの頭上に「!」マーカー+ミニマップに光点を表示する。
A*経路は既にあるので、タップ誘導の光の粒を道に沿って流す演出も可能だが、まずは「!」だけで十分。

**A-3. 操作説明の常設化** 【効果中・工数小】

設定画面(⚙)に「操作説明」項目を追加し、タイトルのhowtoList(1183-1189)と同じ内容
(タッチ/PC自動切替 9615-9621 も再利用)をいつでも見られるようにする。

**A-4. 資質未割り振りバッジ** 【効果小・工数小】

未割り振りポイントがある間、メニューボタン(I)に赤い点を出す。A-1のステップ4と連動。

### B層: 翌日戻ってくる理由を作る(再訪フック)

**B-1. 日替わりの御用(デイリークエスト)** 【効果大・工数中】

高札に「今日の御用」を1件追加。日替わりで討伐対象が変わる(例: ガキ5体/イヌガミ3体/ガシャドクロ3体…)。
報酬はモン+徳+低確率で強化巻物のかけら等。

- 日付判定は既存のJST基準(play.html:1504 `JST_OFFSET_MS`)を使い、JST 0時に更新。全端末で同じ御用
- 受注状態と達成日は セーブに `daily:{date, questId, done}` として保存
- 対象は日付のハッシュで決定論的に選ぶ(サーバー不要。`Date.now()`とJSTオフセットだけで完結)

**B-2. 連続入庁の恩賞(ログインボーナス)** 【効果中・工数小】

入庁時にJST日付が前回と変わっていれば恩賞を配る。連続日数で少しずつ良くする
(1日目 甘露水S → 3日目 韋駄天の水 → 7日目 竜の血1、以降7日周期)。
連続が切れても責めない(1日目に戻るだけ)。閻魔庁の世界観に合わせ「勤怠印」などの名で。

**B-3. 「次の見どころ」表示** 【効果小・工数小】

入庁直後のログに、実時間連動要素の予告を1行出す:
「次の夜は◯時から」「ゴウリュウは討伐から30分で蘇る」など。既存の昼夜サイクル(1561-1562)と
ボス復活仕様(DRAKE_RESPAWN_SECONDS, 1500)の情報を表示するだけ。

**B-4. 更新情報メールの運用開始(Resend)** 【効果中・実装済み基盤の活用】

仕組みは既にある(account.js:771-775 → sync-resend-contact)。足りないのは「送る中身」。

- 新バージョン公開時に更新情報メールを配信する運用を始める(ロードマップのv0.xx項目がそのままネタになる)
- 文面のテンプレート作成と各回の原稿は **Coworkの担当にできる**(push不要な仕事)
- 登録直後のウェルカムメール(遊び方3行+次の目標)もResend側で設定可能

**B-5. PWA導線の補強** 【効果小・工数小】

iOS向け案内(play.html:8890-8906)に加え、Androidで `beforeinstallprompt` を拾って
「ホーム画面に追加」ボタンを設定画面に出す。ホーム画面から起動する人は再訪率が構造的に高い。

## 3. 優先順位と進め方

```
第1週: C層(計測イベント一式) + A-3(ヘルプ常設) + A-4(バッジ)   ← 小さくて確実
第2週: A-1(着任チェックリスト) + A-2(オキナ道標)                ← 初回10分の本丸
第3週: B-2(連続入庁の恩賞) + B-3(見どころ表示)                  ← 再訪の軽いフック
以降 : B-1(日替わり御用)、B-4(メール運用)、B-5(PWA)
       → GA4のファネルデータを見て次を決める
```

判断基準: `game_start → first_kill` の到達率と、GA4の「1日後リテンション」。
チェックリスト導入の前後でこの2つを比べれば、効果が数字で分かる。

## 4. 実装上の注意(AGENTS.mdとの整合)

- 変更はすべてクライアント側(play.html / online/*.js / index.html)。**workerのデプロイは不要**
- online/analytics.js 等を変更したら `?v=` を上げる(現行 `?v=20260723a`)
- セーブ形式を上げる場合(A-1, B-1, B-2)は SAVE_VERSION を16へ。旧セーブの自動移行を必ず書く
  (チュートリアルは達成済み扱い、デイリー/恩賞は未受領初期値)
- デイリー判定はJSTで統一(既存の `JST_OFFSET_MS` を再利用)。端末時計のずれによる多重受領は、
  最後に受領したJST日付をセーブに残す方式で許容範囲に抑える(厳密化はサーバー権威化まで保留)
- イベント追加時は docs/analytics-setup.md のイベント一覧を同時更新
- 完了確認は従来どおり `bash scripts/check_release.sh` 全OKまで

## 付録: GA4側の設定(ユーザー作業・約15分)

1. GA4 →「探索」→「ファネルデータ探索」を新規作成
2. ステップを `game_start` → `first_kill` → `quest_accept` → `zone_enter(zone=cave)` の順で登録
3. 「経過時間を表示」をONにして、各段の所要時間も見る
4. 併せて「維持率」レポートで1日後・7日後リテンションを確認
5. Clarityはトップ・冥職名鑑向け(Canvas内は録れない)なので、ゲーム内分析はGA4イベントに一本化
